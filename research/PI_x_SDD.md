## 为什么需要两套系统？

你有没有经历过这样的场景——需求文档写得漂漂亮亮，代码写到一半发现方案不对；或者更惨——交付了，用户说"这不是我要的"；又或者——上次踩的坑，换个项目又踩了一遍？

这些问题的根源，是**"想"和"做"之间缺了一座桥**。

一个类比：

```
SDD (spec-kit)  =  建筑师画图纸
PI (智行合一)    =  工地总指挥 + 质检 + 安全员 + 经验老师傅

光有图纸，工地没人管 → 施工走样、遇到地基问题没人决策
光有总指挥，没有图纸 → 凭经验干活，需求丢三落四
```

| 你遇到的问题 | spec-kit 能解决吗？ | PI 能解决吗？ |
|------------|:---:|:---:|
| "需求想了半天，写出来还是漏洞百出" | ✅ 模板强制完整 | — 交给 SDD |
| "代码写到一半发现方案不对，已经改了3次了" | ❌ 不管实现过程 | ✅ 战势升阶，自动换策略 |
| "上次踩的坑，这次又踩了" | ❌ 无记忆 | ✅ 鉴簿复用，同坑不踩两次 |
| "交付了，用户说这不是他要的" | ❌ 无交付确认 | ✅ 明约双签 |
| "团队三个 Agent 各干各的，谁也不看谁" | 🟡 格式兼容 | ✅ 战报+决策三权+Coach巡检 |

**结论：SDD 管"想清楚再动手"，PI 管"动手之后别翻车"。两者叠加 = 全生命周期治理。**

> ⚡ **轻量任务不需要 SDD**：修个 typo、写个工具函数、改个配置——直接进 PI 编程四令就够了。SDD 的前置阶段（需求→调研→方案→任务）在这些场景下是不必要的仪式开销。PI 的难度三档（🏊🏻轻量/🏋️标准/🐲深度）自动判定：只有🏋️标准和🐲深度任务才值得启动 SDD。

接下来，我们跟着"小明想做一个记账软件"的故事，走完每一个阶段。

## Phase 1: 需求 —— "我想做个记账 App"

### 小明的需求

> "我想做一个个人记账软件，能记录收支，看月度报表，设置预算提醒。"

### SDD 接管：`/speckit.specify`

SDD 用 spec 模板把小明模糊的想法变成结构化规格：

```markdown
# Feature Specification: 个人记账系统

## User Story 1 - 快速记一笔 (Priority: P1) 🎯 MVP
用户打开 App，30秒内记录一笔收支。
**Independent Test**: 记完一笔后能在列表里看到。
**Acceptance Scenarios**:
1. Given 用户在首页, When 点击"+"输入金额50选择"餐饮", Then 列表显示"餐饮 -¥50"
2. Given 用户输入金额, When 不选分类直接保存, Then 自动归入"其他"
[NEEDS CLARIFICATION: 是否支持语音输入？]

## User Story 2 - 月度报表 (Priority: P2)
...

## User Story 3 - 预算提醒 (Priority: P3)
...
```

### PI 在旁做什么？

PI 此时**不干涉** SDD 的模板工作，但已进入**静默预热模式**——启动三查在后台运行，收集信息但不输出，等 SDD 退场后零切换接管：

```
🔍 查境: 未指定技术栈 → [待澄清: Web/Mobile/Desktop?]
📖 查史: 无相关鉴簿
🎯 查标: 必达=记账+查看 | 应达=报表+预算 | 可达=多账本+导出
```

> **协同点**：SDD 的 `[NEEDS CLARIFICATION]` 和 PI 的信息判别协议在这里**自然合流**——两者都要求"不确定就标出来，不猜测"。

---

## Phase 2: 调研 —— "用什么技术？"

### SDD 接管：`/speckit.plan` (research phase)

SDD 生成 `research.md`：

```markdown
# Research: 个人记账系统

## 前端框架对比
| 框架 | 优势 | 劣势 | 推荐 |
|------|------|------|------|
| React Native | 跨平台，生态成熟 | 性能中等 | ✅ |
| Flutter | 性能好，UI一致 | Dart学习成本 | 🔄 |

## 数据存储对比
| 方案 | 离线支持 | 同步能力 | 推荐 |
|------|---------|---------|------|
| SQLite + 云同步 | ✅ | 需自建 | ✅ |
| Firebase | ✅ | 内置 | 🔄 |
```

### PI 叠加了什么？

PI 识别"选型对比"关键词，进入 📊**产品决策**场景，激活 🧠**最强大脑**认知阵（统帅+建筑师），聚焦「痛点→拆解→评估→数据验证」管线。在这套心智下，**致人术·方案比选**增强 SDD 的调研：

```
📊 方案比选
| 方案 | 成本 | 收益 | 风险 | 推荐 |
| A) React Native + SQLite | 2周 | 跨平台+离线 | 复杂查询性能 | ✅ |
| B) Flutter + Firebase | 1.5周 | 快速上线 | 厂商锁定 | 🔄 |
你最在意哪个维度？（开发速度/离线能力/长期维护...）
```

> **协同点**：SDD 的 research.md 是**持久化文档**（给人看），PI 的致人术·方案比选是**交互式决策**（即时对话）。两者互补：SDD 记录调研过程，PI 引导做出决策。

---

## Phase 3-4: 方案与任务 —— "怎么建？按什么顺序？"

读者已经看出了节奏：SDD 出文档，PI 补判断。方案和任务两个阶段继续这个分工，我们快进——

**SDD 输出 `plan.md`**，包含宪章检查（不做多币种？✅ 不包装 SQLite？✅）和项目结构（models / screens / services / utils 四层）。然后 **`tasks.md`** 把 12 个任务按用户故事分组，标注可并行项和里程碑：

```markdown
## User Story 1 - 快速记一笔 (P1) 🎯 MVP
- [ ] T003 [P] 创建 Transaction model
- [ ] T004 [P] 创建 Category model
- [ ] T005 实现 StorageService (依赖 T003, T004)
- [ ] T006 实现 AddTransaction screen
- [ ] T007 实现 Home screen (交易列表)
**Checkpoint**: P1 完成 = MVP 可用! ✅

...（后续 P2 月度报表、P3 预算提醒，共 12 个任务）
```

**PI 在方案落地前做两件事**——

一是**约束锚定**，把核心约束钉死，后续编码不可漂移：

```
💭 链: 观(记账App·React Native)
     →析(核心约束: 离线优先·SQLite·单用户)
     →策(Model层先行·Service层隔离·Screen最后)
     →验(每层完成后独立测试)
```

二是**并行安全检查**，验证 SDD 标了 `[P]` 的任务是否真的不冲突：

```
排·依赖: T003[P] + T004[P] ✅ 不同文件，可并行
         T005 依赖 T003+T004 ✅ 串行
```

PI 判定为 🏋️标准难度（12 任务跨多文件），自动建立任务追踪。SDD 的 spec / plan / tasks 与 PI 的 需求.md / 方案.md / 任务.md 天然一一对应——有 SDD 时用 SDD 制品，没有时 PI 自己兜底。

> **一句话**：SDD 定义**结构**（"建 3 层"），PI 定义**纪律**（"建好一层验一层"）。SDD 的 tasks 是**静态计划**，PI 把它变成**动态战域**——记录进度、标记失败、调整顺序。

## Phase 5: 编码 —— "开干！"

### 🔑 从这里开始，SDD 退场，PI 全面接管

SDD 的工作到 tasks.md 就结束了。接下来的编码、调试、交付全部由 PI 治理。

**这就是为什么需要两套系统**——spec-kit 单独使用时，从这一步开始就"裸奔"了。

PI 接管的第一个动作，不是写代码——而是**激活编程场景**：

```
🧠 PI · 🖥️编程开发 · 🧠最强大脑(统帅+建筑师) · 💡 本质→正名→实现→验证 · 🏋️标准
```

这行"场景公示"背后是 PI 的认知路由系统——九大场景，每个场景激活不同的**认知阵**（心智模式组合）。编程场景激活的是"统帅+建筑师"：**统帅**锚定目标、预判风险，**建筑师**洞察本质、规划结构。接下来的编程四令、正名三则、步步为营，都在这套心智下运行。

> 等到后面遇到性能问题进入调试时，你会看到 PI 切换为完全不同的心智——这种"按需切换大脑"的能力，是 PI 区别于普通 AI 助手的核心之一。

> **回退安全网**：如果 PI 执行过程中发现 spec 本身有根本性设计缺陷（不是 bug，是需求/方案层面的错误），PI 会通过谏言协议向用户报告，建议回退 SDD 重新 specify/plan。已完成的代码通过步步为营固化，不丢失。

### PI 依赖治理——第一个任务就藏着雷

第一个任务 T002 就是"配置 SQLite + TypeORM"——看起来人畜无害？这是典型的**新增依赖**场景，生产项目中"升级依赖炸了全站"的事故比比皆是。PI 的依赖治理协议自动启用：

```
T002: 配置 SQLite + TypeORM

一·查变更: TypeORM 0.3.x CHANGELOG → ⚠️ 0.3 移除了 ormconfig.json，改用 DataSource
二·扫影响: 新增依赖，无存量引用
三·验兼容: npm install → build → test ✅
四·定回退: 万一不兼容 → 回退 typeorm@0.2 + 战域卡记录原因
```

四步走完，风险拦在 `npm install` 之前，不等上线才发现。

### PI 编程四令

```
开始 T003: 创建 Transaction model

💭 链: 析·本原(Transaction: id/amount/category/date/note)
     →锚·约束(SQLite·TypeORM·离线优先)
     →校·命名(Transaction不叫Record，避免与JS保留词冲突)
     →定·验收(model可实例化+可序列化+可持久化)
```

写到 StorageService 时，小明随手把保存方法命名为 `save()`。PI 的**正名三则**介入——

```
📏 正名: save() 太泛——保存的是 Transaction，应叫 saveTransaction()
   └── 理由: BudgetService 也需要保存，两个 save() 组合使用时调用方不知道在存哪张表
```

看起来吹毛求疵？两周后写 BudgetService 时，小明庆幸当初没有偷懒。

### PI 步步为营

```
T003 ✅ → git commit "feat: add Transaction model"
T004 ✅ → git commit "feat: add Category model"
T005 ✅ → git commit "feat: implement StorageService"
         └── 💭 排: T003/T004 model 与 T005 service 集成测试通过
```

每完成一个任务就 commit，固化战果。

### PI 验证矩阵

| 任务 | 变更类型 | 验证方式 | 通过？ |
|------|---------|---------|:---:|
| T002 | 依赖变更 | install + build + test | ✅ |
| T003 | 代码逻辑 | build + test | ✅ |
| T005 | 数据/Schema | migrate + 数据校验 | ✅ |
| T006 | UI | build + deploy + 人工视觉确认 | ✅ |

> **关键差异**：spec-kit 在这个阶段完全帮不上忙。PI 的编程四令+步步为营+验证矩阵 = spec-kit 没有的"施工纪律"。
>
> **场景链跳转**：Phase 5（编码）和 Phase 6（调试）不是线性的——它们构成 🖥️→🔧→🖥️ 循环。每次跳转不只是换标签，是**整套认知阵的切换**：编码的"统帅+建筑师"⇄ 调试的"分析师+守卫"。PI 的场景路由自动处理跳转，连心智模式都一并切换。

### PI 反模式预检

编码过程中，PI 的反模式十戒默默守护。比如小明在写 StorageService 时想"应该用 SQLite 的 WAL 模式"——

```
🚫 猜而不搜! "应该用WAL模式" → 先搜索 TypeORM + SQLite WAL 兼容性
💭 排: 搜索后确认 TypeORM 默认已启用 WAL → 无需额外配置
```

十戒不是事后检查，而是**内化的行为约束**——每一步都在防止"猜而不搜""改而不验""重而不换"。

T003 到 T007，一路绿灯。每完成一个任务就 commit，测试全过。小明开始觉得——"有 PI 看着，写代码也太丝滑了。"

他灌了 1000 条测试数据，准备做 T008 的 ReportService——月度报表。

## Phase 6: 遇到麻烦 —— "怎么这么卡！"

月度报表页面加载——8 秒。小明盯着转圈的加载动画，眉头拧了起来。

PI 检测到"性能/卡顿"信号，场景自动切换——

```
🔗 PI · 🖥️编程开发 → 🔧调试排障 · 🔬精密验证(分析师+守卫) · 💡 读败→定界→溯源→验假→固防
```

注意认知阵的变化——从编码时的"统帅+建筑师"切换为"分析师+守卫"。**分析师**负责逻辑深挖、穷源竟委（"到底为什么慢？"），**守卫**负责经验标准、规范执行（"修完别引入新问题"）。这不是换了个标签，是 PI 字面意义上**换了一颗脑子**。

spec-kit 的 tasks.md 上只写着 `T008 实现 ReportService`——至于性能问题怎么办？不在它的管辖范围。

PI 在新的心智模式下，**调试六步自动启动**：

```
步一·读败: 月度报表查询耗时 8s，目标 <500ms
步二·定界: ReportService.getMonthlyReport()
步三·溯源: SELECT * FROM transactions WHERE date BETWEEN ...
           └── 无索引！全表扫描 1000 条
步四·比对: Home 列表查询 <100ms（只取最近 20 条，有 LIMIT）
步五·验假: 假设: 加索引后 <500ms
步六·固防: ✅ 加 date 索引 → 查询 120ms → 加回归测试
```

```
📝 已试: ✅加索引→解决 | ⚡下策: 继续 T009 报表 UI
```

### 如果问题更复杂呢？

小明加了索引，满怀信心地刷新页面——3 秒。

数据量到了 10 万条，索引也救不了聚合查询。小明脸上的笑容凝固了。

但这次不一样——PI 检测到连败 2 次，战势自动升阶：

```
🧠 PI · 战势·⚡易辙 · 肃阵

战况：索引优化后 10 万条仍需 3s，连败2次
情报：✅ 已证: 瓶颈在聚合计算 ❌ 已排: 索引问题 🔍 未锁: 预计算 vs 分页
损益：续战(预计0.5天可解) vs 止损(降级为分页加载)
新策：预计算月度汇总表（写入时更新），查询走汇总表

📉 败因: #聚合瓶颈 · 大数据量实时聚合不可行，需预计算
🌐 路径: A)预计算汇总(0.5天) B)分页加载(0.3天) C)后台计算+缓存(1天)
```

注意 PI 在升阶时自动执行**认知防护四则**：
- **沉没成本归零**——不因"已经在索引方案上花了很多时间"而继续死磕
- **约束再锚定**——重读五敕令+战域卡，确认约束没有漂移
- **失败分类**——"加索引"和"调SQL"是同一根因（查询层面），只计1次失败
- **回归意识**——修复后验证上下游未被破坏

> **这就是 PI 的核心价值**：spec-kit 不知道你遇到了问题，不知道你试过什么，不知道该换什么策略。PI 的战势六阶 = 一个永远冷静的"经验老师傅"——该换策略时绝不死磕，该止损时绝不恋战。

### PI 止损三阶——"越挫越勇"的另一面

战势六阶管"越挫越勇"（攻），但 PI 还有另一套并行机制——**止损三阶**管"量入为出"（守）：

```
🟢 正常: 标准探索，直接执行
🟡 预警: 连续 3+ 次失败或已执行九令中 ≥5 令 → 主动告知消耗，建议是否继续
🔴 止损: 九令洞鉴完成仍未解 → 善始善终交接，不强行消耗
```

在我们的例子中：索引方案失败后，PI 同时做了两件事——**战势升到⚡易辙（换预计算策略）** + **止损评估仍在🟢（才2次失败，远未到🟡预警线）**。如果这个问题再失败 3 次，PI 不会继续死磕——它会主动告诉小明"已消耗 X 个方案，建议是否继续"。

> **战势管攻，止损管守——二者并行互补，构成容错闭环。** 没有止损的战势 = 盲目死磕；没有战势的止损 = 过早放弃。

### PI 识·认知健康度——"我的判断还靠谱吗？"

调试复杂问题时，还有一个隐患比"方案不对"更危险——**PI 自己的规则体系出现矛盾**。比如鉴簿说"大数据量用预计算"，但当前约束是"写入延迟必须 <10ms"，两者可能冲突。如果 PI 没有意识到这种冲突，就会在两个方向之间摇摆，输出互相矛盾的建议。

PI 的**明心·识**机制专门监控这种"认知健康度"：

```
🧠 PI状态: 信🟡中(1证据:日志显示超时但未定位根因) · 量🟢充裕(已试1/6策) · 识🟢自洽
  → 补充网络拓扑信息可升至🟢。下一步: 抓包分析请求链路。
```

这行状态报告中，`识🟢自洽` 就是认知健康度的实时读数：

| 识 | 含义 | AI 行为 |
|---|------|---------|
| 🟢 自洽 | 规则体系内无矛盾 | 正常推进 |
| 🟡 存疑 | 两处规则对类似场景给出不同建议 | 标记冲突点，继续但关注，提醒用户 |
| 🔴 矛盾 | 规则互斥，影响决策 | **强制减法**——停止写入新规则，复查冲突，请求人类裁决 |

在我们的例子中：如果 PI 发现鉴簿的"预计算"经验与当前项目的"低写入延迟"约束冲突，识从🟢降至🟡，PI 会主动标出冲突点："鉴簿建议预计算（写入时更新汇总表），但当前约束要求写入 <10ms，两者可能冲突。建议评估写入增量。"——不是默默选一个，而是让人类裁决。

> **识的核心哲学**：PI 不追求"规则越多越好"，而是"在有限的规则下保持自洽"。当认知熵增失控时，做减法而不是加法。这种"自我监控"能力，是 PI 区别于"堆砌规则"的简单 prompt 的关键。

🔒 校·闭环: 交付路径包含验证步骤（npm test已执行·结果已附）

### PI 交付六令

```
✅ 验证: npm test → 42 tests passed, 0 failed
🔎 核验: Transaction CRUD + 月度报表 + 预算提醒全链路通过

### PI 明约确认——交付不是单方面宣布

交付不是 AI 说一句"做完了"就完了——明约要求**双向确认**：

```
📋 PI: 交付确认——
  "30秒记一笔" → ✅ 首页一键记账
  "月度报表"    → ✅ 图表+分类统计

> **小明**：等等——预算提醒能按分类设吗？比如"餐饮不超过 2000"？

```

   当前版本 = 总支出超过月度预算时通知。确认交付？
```

这段对话展示了三个 PI 能力：**明约**确保交付物与需求对齐，**致人术**引导用户区分"当前版本"和"未来需求"，**谏言协议**在发现范围蔓延时及时刹车。

### PI 致人术·交付后排查

明约确认后，PI 不是"交完了就完了"——致人术的同类排查和关联预判自动启动：

```
🔍 同类排查: 其他 Service 是否有类似的大数据量查询？
   └── BudgetService.checkOverBudget() → 也用了 SUM 聚合 → 主动应用同一预计算策略（不只报告，直接修复）
🔗 关联预判: 预计算汇总表引入后，Transaction 的 INSERT 触发器是否影响写入性能？
   └── 测试: 批量插入 1000 条 → 耗时从 0.3s 增至 0.5s → 可接受
```

> **致人术**的核心是"致人而不致于人"——主动出击排查隐患，不等 bug 找上门。

### PI 战后三省

交付完成不是终点，PI 会做一次经验复盘：

  困·隘: SQLite 大数据量聚合查询——10万条实时聚合不可行

  功·长: 调试六步定界+溯源能力↑，战势升阶自然切换策略

📌 升格评估: "移动端SQLite预计算"为元知识候选
   → 边界清晰(移动端+SQLite+>1万条) → 转入鉴簿待跨任务验证

> **这一步是 PI 经验飞轮的起点**——三省产出的📌候选，经升格五问评估后转入鉴簿，为未来同类项目提供免费的"经验老师傅"。

### 不同平台上的 PI——Loop 与 Auto

小明先在 Copilot CLI（按请求计费）上开发，后来切到 Claude Code（按 token 计费）继续。同样的 PI，行为有微妙差异：

| | 🔄 Loop 模式（按请求计费） | ⚡ Auto 模式（按 token 计费） |
|---|---|---|
| 典型平台 | Copilot CLI、OpenClaw | Claude Code、Cursor |
| 核心行为 | 每次输出**必须以提问收尾**——会话终止 = 用户被扣费 | AI 自主决定交互节奏，简单任务静默交付 |
| 仪式密度 | 全量输出，一次请求内完成更多轮交互 | 按需输出，轻量任务跳过仪式 |
| 交付后 | "还有什么需要调整的吗？" | 直接结束（🏊🏻）或追问（🏋️🐲） |

**为什么这很重要？** 同样是做完一个功能，在 Copilot CLI 上 PI 会说"功能完成，要我跑一下测试吗？"——因为如果不提问就沉默，会话就终止了，用户得重新付费开一轮。而在 Claude Code 上，PI 可能直接跑完测试、commit、然后安静结束——因为 token 计费下，不必要的对话反而是浪费。

> **PI 不是一成不变的脚本**——它感知自己运行的平台环境，自动调整交互策略。这种"平台适应性"让同一套方法论在不同工具链上都能发挥最大价值。
