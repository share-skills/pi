## 一、为什么需要可视化？

### 1.1 AI 决策的"黑箱"困境

你每天都在用 AI 编程，但有没有遇到过这样的场景？

- **失败了，不知道哪步错**：AI 给了你一个方案，跑不通。你问它为什么，它又给一个完全不同的方案——你依然不知道问题出在哪。
- **成功了，不知道怎么复现**：这次 AI 帮你完美解决了一个复杂 bug，但下次遇到类似问题，你不确定该怎么引导它再来一次。
- **团队协作，无法对齐**：你的同事说"AI 帮我重构了整个模块"，但你无法理解它为什么那样拆分，也无法评估方案的优劣。
- **成本不透明**：一个会话跑了 30 分钟，消耗了多少 token？值不值得？下次能不能更高效？

这就是"黑箱"困境——AI 在你看不见的地方做出一连串决策，你只能看到最终结果。就像医生只告诉你"吃这个药"，却不告诉你为什么、其他方案有哪些、风险在哪里。

### 1.2 可视化带来的改变

如果把黑箱变成水晶球呢？

| 黑箱模式 | 水晶球模式（可视化） |
|---------|-------------------|
| AI 给了答案，不知道为什么 | 看见每个决策点：场景判断 → 策略选择 → 信心评估 |
| 失败了不知道原因 | 追踪战势升级：从"简单调整"到"系统级重构"的完整路径 |
| 无法评估 AI 效能 | 量化指标：token 消耗、复杂度评分、成功率一目了然 |
| 团队复盘靠"感觉" | 导出决策记录，用数据说话，用更强的模型分析和改进 |

可视化不是锦上添花——它是让你从"使用 AI"变成"掌控 AI"的关键一步。

### 1.3 一个真实的例子

想象你正在调试一个 WebSocket 内存泄漏问题。没有可视化时，你的体验是这样的：

> "AI 看了半天代码，试了三次都没修好，第四次突然改了一大堆文件，居然好了。我不知道前三次为什么失败，也不知道第四次到底改对了什么。"

**有了 PI 决策可视化，你看到的是完整的决策图谱：**

```
🟣 编程开发·场景启动
 └─🟢 第1次尝试：鹰(🦅)前瞻，检查连接管理  →  ❌ 失败
    └─🟡 战势升级 → L1·易辙：换视角
       └─🟢 第2次尝试：检查事件监听器清理  →  ❌ 失败
          └─🔴 战势升级 → L2·深搜：鲨(🦈)深度搜索
             └─🟢 第3次尝试：全局引用链分析  →  ❌ 失败
                └─🟣 战势升级 → L3·系统：龙(🐲)系统洞察
                   └─🟢 第4次尝试：发现闭包持有 socket 引用
                      → 重构生命周期管理  →  ✅ 成功！
```

现在你清楚地看到：
- **第 1-2 次失败**是因为只看了表层（连接管理、事件监听器）
- **第 3 次**扩大了搜索范围但方向仍不够深
- **第 4 次**切换到🐲龙（系统洞察），用全局视角发现了闭包引用——这才是根因

下次遇到类似问题，你就知道：**内存泄漏应该直接用系统级视角，而不是从表层开始一步步试**。这就是可视化的力量：把经验从"隐性"变成"显性"。

## 二、PI 决策可视化能做什么？

### 2.1 核心能力一览

- 📊 **决策图谱**：树形结构展示所有决策节点和它们的因果关系，支持拖拽、缩放
- 🕐 **时间线回放**：滑块控制，按时间顺序重现 AI 的完整思考过程
- 📈 **效能指标**：token 消耗、复杂度评分、战势级别、灵兽激活——底栏实时更新
- 🔍 **详情面板**：点击任意节点查看完整上下文——策略、典故、信心、模型信息
- 📤 **一键导出**：隐私保护的 JSON 导出和单文件 HTML 分享
- 🖥️ **实时监控**：WebSocket 推送，边工作边看 AI 的决策过程
- 👥 **多 Agent 追踪**：Leader → Teammate → Coach 的完整交互链可视化

### 2.2 支持的平台

PI 决策可视化通过 hooks 机制自动采集数据，支持所有主流 AI 编程工具：

| 平台 | 集成方式 | 数据采集 |
|------|---------|---------|
| Claude Code | 原生 `hooks.json` 配置 | ✅ 自动 |
| GitHub Copilot CLI | `install.sh` 一键安装 | ✅ 自动 |
| Cursor | `.cursorrules` 集成 | ✅ 自动 |
| Kiro | 规则文件集成 | ✅ 自动 |
| 其他工具 | `capture-generic.sh` 脚本 | 📝 手动/脚本 |

### 2.3 数据是如何被采集的？

PI 通过 **7 种事件钩子（hooks）** 自动捕获决策数据，你无需做任何额外配置：

| 钩子事件 | 触发时机 | 捕获脚本 | 记录内容 |
|---------|---------|---------|---------|
| `UserPromptSubmit` | 你输入提示词时 | `capture-prompt.sh` | 输入摘要、字数、交互模式 |
| `PostToolUse` | 工具执行成功后 | `capture-tool-result.sh` | 工具名、退出码、耗时、输出摘要 |
| `PostToolUseFailure` | 工具执行失败后 | `capture-tool-result.sh` | 失败原因、错误输出、退出码 |
| `Stop` | 会话结束时 | `capture-decision.sh` | 完整 PI 状态（场景、战势、灵兽、策略、信心） |
| `SubagentStart` | 子 Agent 启动时 | `capture-subagent.sh` | Agent ID、角色、任务 |
| `SubagentStop` | 子 Agent 结束时 | `capture-subagent.sh` | Agent ID、结果、耗时 |
| `PreCompact` | 上下文压缩前 | `pre-compact.sh` | 状态快照（防止压缩丢失关键信息） |

所有数据以 JSONL（每行一条 JSON）格式存储在本地：

```
~/.pi/decisions/
├── 2026-01-15/
│   ├── session-abc123.json              ← 会话元数据
│   ├── session-abc123.events.jsonl      ← 事件流
│   └── session-abc123.nodes.jsonl       ← 决策节点
├── 2026-01-16/
│   └── ...
└── ...
```

**隐私保护**：所有数据在写入前经过 `sanitize_text()` 处理——自动脱敏文件路径、密钥、长 token 等敏感信息。你的代码和凭据不会出现在决策记录中。

## 三、快速开始：5 分钟上手

### 3.1 安装

```bash
# 方式一：随 PI 一起安装（推荐，自动配置所有 hooks）
curl -fsSL https://raw.githubusercontent.com/share-skills/pi/main/install.sh | bash

# 方式二：独立安装可视化工具
curl -fsSL https://raw.githubusercontent.com/share-skills/pi/main/scripts/setup-standalone-visualize.sh | bash
```

安装过程是交互式的——箭头键选择平台，空格键切换，回车确认。安装完成后，hooks 会自动配置到你选择的 IDE 中。

### 3.2 第一次使用

```bash
# 步骤 1：在支持的 IDE 中正常使用 PI
# （写代码、调试、提问...所有交互都会被自动捕获）

# 步骤 2：打开可视化界面
/pi visualize
# 或者使用独立命令
~/.pi/visualize.sh
```

浏览器会自动打开 `http://localhost:3141`，你就能看到完整的决策图谱了。

### 3.3 界面概览

```
┌──────────────────────────────────────────────────┐
│ ⚡ PI Visualizer   [◀ ▶ Timeline ═══●════]  指标 │  ← 顶栏：时间线滑块 + 播放控制
├─────────┬────────────────────────┬───────────────┤
│ 📅 日期  │                        │  [详情面板]   │
│ 📋 会话  │     决策画布            │  场景/灵兽    │
│   列表   │     (可拖拽/缩放)       │  策略/典故    │
│         │                        │  信心/模型    │
├─────────┴────────────────────────┴───────────────┤
│ tokens: 12.4k │ 复杂度: 3/5 │ 质量: ⭐⭐⭐⭐ │ ✅ │  ← 底栏：效能指标
└──────────────────────────────────────────────────┘
```

- **左侧边栏**：按日期分组的会话列表，绿色圆点 = 活跃会话，灰色 = 已结束
- **中央画布**：React Flow 驱动的决策节点图，支持拖拽、缩放、节点展开/折叠
- **右侧面板**：点击任意节点后展示完整决策详情
- **顶栏滑块**：拖动或播放，按时间顺序重现整个决策过程
- **底栏指标**：token 消耗、复杂度评分、质量评分、当前状态

## 四、深入理解：决策节点详解

### 4.1 一个决策节点包含什么？

每个决策节点是 AI 在某个时刻的完整"思维快照"。以下是关键字段：

**场景（Scene）** — AI 当前在做什么？

PI 有 9 大场景，每个场景对应不同的认知模式和处理管线：

| 场景 | 认知阵 | 处理管线 |
|------|-------|---------|
| 🖥️ 编程开发 | 🧠 最强大脑（统帅+建筑师） | 本质 → 正名 → 正合实现 → 实证验证 |
| 🔧 调试排障 | 🔬 精确验证（分析师+守护者） | 读败 → 定界 → 溯源 → 验证 → 固防 |
| 🧪 测试质保 | 🔬 精确验证（分析师+守护者） | 定义 → 设计 → 执行 → 分析 → 固防 |
| 📊 产品决策 | 🧠 最强大脑（统帅+建筑师） | 痛点 → 拆解 → 评估 → 数据验证 |
| 📈 运营增长 | 🎯 增长飞轮（统帅+探索者） | 目标 → 实验 → 度量 → 迭代 |
| 🎨 创意发散 | 🌊 创新引擎（建筑师+探索者） | 自由态 → 过滤 → 萃取 → 结构化 |
| 🤝 用户交互 | 🌙 深度共情（和谐者+探索者） | 展开 → 正义 → 韧性 → 共情 |
| 👥 团队协作 | 🧠 最强大脑（统帅+建筑师） | 角色 → 系统 → 节奏 → 韧性 |
| 💛 情感陪伴 | 🌙 深度共情（和谐者+探索者） | 慈悲 → 温柔 → 觉察 → 韧性 |

**灵兽（Spirit Beast）** — AI 当前的"精神状态"

12 种认知原型，代表 AI 在不同困境下的应对策略：

| 灵兽 | 精神 | 认知功能 | 危难信号 |
|------|------|---------|---------|
| 🦅 鹰 | 洞察 | O(n²) → O(n) 降维 | 埋在细节里，需要俯瞰全局 |
| 🐺🐯 狼虎 | 直言 / 揭伪 | 消除确认偏差 + 暴露沉默风险 | 未验证假设 / 已知风险未说 |
| 🦁 狮 | 斗志 | 突破局部最优 | 即将放弃，需要背水一战 |
| 🐎 马 | 速度 | 收紧时间约束 | 效率低下，需要即验即交 |
| 🐂 牛 | 韧性 | 不剪枝的搜索 | 苦差事，需要坚韧以行 |
| 🦈 鲨 | 搜索 / 深搜 | 最大化信息增益 + 深层风险检测 | 猜而不搜 / 浅搜避深 |
| 🐝 蜂 | 并行突击 | 并行执行 + 信息共享 | 终极冲刺，全原型协同 |
| 🦊 狐 | 审慎 | 元认知验证 | 产出质量不达标 |
| 🐲 龙 | 极致 | 全力资源投入 | 极限突破或坦诚边界 |
| 🦄 独角兽 | 卓越 | 可行 → 最优 | 草率交付，需精益求精 |
| 🦉 猫头鹰 | 清明 | 激活深度思考 | 过早下结论 |
| 🐬 海豚 | 流变 | 跨域类比搜索 | 思维僵化，需触类旁通 |

**战势级别（Battle Level）** — 失败后的升级路径

| 失败次数 | 级位 | 主导灵兽 | 策略转变 |
|---------|------|---------|---------|
| 2 次 | ⚡ 易辙 L1 | 🦅 鹰 | 换视角，换方法 |
| 3 次 | 🦈 深搜 L2 | 🦈 鲨 | 根因分析，穷举搜索 |
| 4 次 | 🐲 系统 L3 | 🐲 龙 | 全局庙算，九阶排查 |
| 5 次 | 🦁 决死 L4 | 🦁 狮 | 最小 PoC + 隔离 + 替代路线 |
| 6 次 | ☯️ 截道 L5 | 全原型 | 反向推理 + 跨域类比 |
| 7+ 次 | 🐝 天行 L6 | 🐝 蜂 | 全认知原型轮转 + 外部信息 |

**其他关键字段：**

- **策略（Strategy）**：孙子兵法战术，如"以正合以奇胜""求之于势""投之亡地"
- **信心强度（Confidence）**：高 / 中 / 低——AI 对当前方案的自评
- **典故（Classical Reference）**：触发的经典智慧，如"兵无常势，水无常形"
- **认知模式（Cognitive Mode）**：ENTJ-统帅 / INTJ-建筑师 / INTP-分析师 / ISFJ-守护者 / ENTP-探索者 / INFJ-和谐者
- **模型上下文（Model Context）**：模型名称、输入/输出 token 数、估算成本

### 4.2 节点颜色含义

在决策画布上，节点边框颜色由**类别（category）**决定，结果状态以独立徽章显示：

| 边框颜色 | 类别 | 典型内容 |
|---------|------|---------|
| 🔵 蓝色 | `exec` | 工具执行（bash、文件编辑等） |
| 🟣 紫色 | `battle` | 战势升级点（策略切换） |
| 🟢 绿色 | `delivery` | 会话完成、最终交付 |
| 🟡 黄色 | `interaction` | 人类输入、提示词提交 |
| 🟠 橙色 | `team` | 子 Agent 启动/停止 |
| ⚪ 灰色 | `external` | 外部事件捕获 |

### 4.3 边的含义

节点之间的连线表示决策的因果关系：

- **实线** → 顺序流转（上一步完成后进入下一步）
- **虚线** → 失败或跳过的转换
- **粗橙线** → 战势升级转换（级别提升）

## 五、实战场景：从编程到团队协作

### 5.1 场景一：调试一个复杂 Bug

**情境**：Node.js 服务在高并发下出现随机 502 错误。

**决策图谱展示的过程**：

```
🟣 调试排障·场景启动
 │  认知阵: 🔬精确验证 | 灵兽: 🦅鹰
 │
 ├─🔵 读取错误日志 (PostToolUse)
 │   └─ 输出摘要: "ECONNRESET in upstream, 分布在不同时间点"
 │
 ├─🟢 第1次尝试: 检查连接池配置 → ❌ 配置正常
 │
 ├─🟡 战势升级 → L1·易辙 | 灵兽: 🦅→🐬
 │   └─ 典故: "兵无常势，水无常形" — 换个角度看
 │
 ├─🔵 跨域类比: 检查 DNS 解析 (🐬海豚·流变)
 │   └─ 发现: DNS TTL 过期后重新解析导致短暂连接中断
 │
 └─🟢 修复: 启用 DNS 缓存 + 连接预热 → ✅ 成功
     └─ 信心: 高 | tokens: 8.2k | 复杂度: 3/5
```

**复盘价值**：可视化清晰显示，🐬海豚（跨域类比）的介入是关键——它让 AI 从"应用层"跳到"网络层"思考。下次遇到间歇性 502，可以直接从网络层切入。

### 5.2 场景二：多 Agent 协作开发

**情境**：用 PI Agent Team 重构一个大型模块。Leader 分配任务给多个 Teammate。

**决策图谱展示的交互链**：

```
🟣 Leader: 团队协作·场景启动
 │  制定重构计划，分解为 3 个子任务
 │
 ├─🔵 SubagentStart: Teammate-A (数据层重构)
 │   ├─🟢 读取现有 schema → 设计新 schema
 │   ├─🟢 编写迁移脚本 → ✅ 通过测试
 │   └─🟢 SubagentStop: 成功
 │
 ├─🔵 SubagentStart: Teammate-B (API 层适配)
 │   ├─🟢 更新接口定义
 │   ├─🔴 集成测试失败 → 战势升级 L1
 │   ├─🟢 修复类型不匹配 → ✅ 通过测试
 │   └─🟢 SubagentStop: 成功
 │
 └─🟣 Leader: 合并验证
     ├─🟢 运行全量测试 → ✅ 全部通过
     └─🟢 生成重构报告 → 交付
```

**复盘价值**：你可以看到 Teammate-B 遇到了集成问题并自行解决。如果团队复盘发现 Teammate-B 的问题可以通过更好的接口定义避免，就可以优化 Leader 的任务分解策略。

### 5.3 场景三：用导出数据改进 PI

**情境**：你积累了一周的决策记录，想分析 PI 在你的项目上表现如何。

**操作步骤**：

```bash
# 步骤 1：在可视化界面中选择要分析的会话，点击"导出 JSON"
# （导出文件自动脱敏，安全可分享）

# 步骤 2：把导出的 JSON 交给更强的模型分析
# 例如用 Claude Opus 或 GPT-5：
```

> **给 AI 的分析提示词**：
>
> "这是我使用 PI 智行合一引擎一周的决策记录。请分析：
> 1. 哪些场景的成功率最高/最低？
> 2. 战势升级最常发生在什么类型的问题上？
> 3. 哪些灵兽组合最有效？
> 4. 有没有可以优化的决策模式？"

**分析结果示例**：

> - 调试场景中，直接使用🦈鲨（深搜）比从🦅鹰开始更高效
> - 编程场景中，复杂度 > 3 的任务建议一开始就激活🐲龙
> - 你的项目中，类型相关的 bug 占失败的 40%，建议加强类型检查策略

根据这些发现，你可以调整 PI 的策略权重，让它在你的项目上表现更好。

### 5.4 场景四：团队复盘

**情境**：团队周会上，用可视化数据代替"感觉"来复盘。

**操作方式**：

1. 每个成员导出本周的决策记录
2. 在可视化界面中逐个加载，对比决策模式
3. 重点关注：
   - 谁的战势升级次数最多？是问题本身难，还是初始策略不对？
   - 哪些灵兽组合在团队中被频繁激活？说明项目的核心挑战是什么
   - 成功的决策链有什么共同点？提取为团队的"最佳实践"
4. 标注最佳决策链，作为新成员的培训材料

## 六、进阶用法：实时监控与多项目管理

### 6.1 实时监控模式

在一个终端开启可视化，另一个终端正常工作。AI 的决策节点会实时出现在画布上：

```bash
# 终端 1：启动可视化（默认即为实时模式）
~/.pi/visualize.sh

# 终端 2：正常使用 PI 工作
/pi 编程
# 你会看到决策节点实时出现在画布上！
```

**技术原理**：服务端使用 `chokidar` 监听 `~/.pi/decisions/` 目录的文件变更，通过 WebSocket 实时推送新事件到前端。前端 React Flow 自动布局新节点。

**WebSocket 消息类型**：

| 消息类型 | 触发时机 | 内容 |
|---------|---------|------|
| `connected` | 连接建立 | 数据源目录、会话数量 |
| `archive` | 初始加载 | 完整决策存档 |
| `session_update` | 会话变更 | 更新后的会话数据 |
| `new_event` | 新事件到达 | 单个决策节点 |

### 6.2 多项目管理

如果你在多个项目中使用 PI，可以指定不同的数据源：

```bash
# 查看项目 A 的决策记录
~/.pi/visualize.sh --source /path/to/project-a/.pi/decisions

# 查看项目 B 的决策记录
~/.pi/visualize.sh --source /path/to/project-b/.pi/decisions --port 3142
```

### 6.3 会话管理

- **活跃会话**：左侧边栏中绿色圆点标记，表示数据仍在更新
- **已结束会话**：灰色圆点标记，可以随时回放
- **删除会话**：在界面中删除不需要的会话，或直接清理 `~/.pi/decisions/` 下的旧目录
- **会话切换**：使用时间线上的 Prev/Next 按钮，或直接点击左侧列表

### 6.4 键盘快捷键

| 键 | 功能 |
|---|------|
| `[` | 切换左侧边栏（会话列表） |
| `]` | 切换右侧详情面板 |
| `↑` `↓` | 上/下切换会话 |
| `Space` | 播放 / 暂停时间线回放 |
| `?` | 显示帮助面板 |
| `Esc` | 关闭当前面板 |

### 6.5 导出与分享

**导出 JSON**（隐私保护）：
- 点击顶栏导出按钮，自动生成脱敏后的 JSON 文件
- 敏感信息（路径、密钥、长 token）已被替换为 `[REDACTED]`
- 可以安全地分享给团队成员或用于 AI 分析

**导入记录**：
- 点击顶栏导入按钮，选择之前导出的 JSON 文件
- 可视化界面会直接加载并展示，无需手动复制文件

## 七、最佳实践与技巧

### 7.1 养成"可视化复盘"的习惯

**每日 5 分钟复盘**：

1. 打开可视化界面，浏览今天的会话
2. 关注战势升级的模式：是否总在同一类问题上反复失败？
3. 对比成功和失败的决策链，找出关键差异
4. 问自己：如果下次遇到类似问题，可以直接从哪一步开始？

**示例复盘清单**：

```
□ 今天战势升级了几次？最高到哪一级？
□ 哪个灵兽被激活最多？说明什么？
□ 有没有"从鹰直接跳到龙"的情况？值不值得？
□ token 消耗最高的会话做了什么？能否优化？
```

### 7.2 用数据驱动 PI 优化

**三步优化法**：

1. **导出**：选择高复杂度（4/5 以上）的会话，导出决策记录
2. **分析**：用 Claude Opus 或 GPT-5 分析决策模式，识别低效环节
3. **调整**：根据分析结果，调整你与 PI 的交互方式——比如更精确的问题描述、更早地提供上下文、在特定场景下指定灵兽

**常见优化发现**：

- 如果"猜而不搜"频繁出现 → 提供更多上下文给 AI
- 如果战势频繁升到 L3+ → 问题描述可能不够清晰，尝试拆解
- 如果🦊狐（审慎）频繁激活 → AI 对输出质量不够自信，检查验收标准是否清晰

### 7.3 团队协作建议

- **建立共享记录仓库**：将脱敏后的决策记录存入 Git 仓库，方便团队成员查阅和学习
- **定期团队复盘**：用可视化数据代替主观感受，让复盘更有针对性
- **标注最佳实践**：在特别成功的决策链上做标记，作为新成员的培训案例
- **标准化场景入口**：团队约定在不同类型的任务中使用统一的 PI 场景入口

## 八、常见问题

### Q: 我的数据安全吗？

**A:** 所有数据默认存储在本地 `~/.pi/decisions/`，不会上传到任何外部服务。导出时自动脱敏——`sanitize_text()` 会替换文件路径（`/Users/xxx` → `[REDACTED]`）、密钥（`--token=xxx` → `--token=[REDACTED]`）、超过 20 字符的长字符串等敏感信息。你完全控制数据的存储和流向。

### Q: 支持哪些 IDE / 工具？

**A:** Claude Code、GitHub Copilot CLI、Cursor、Kiro 等主流 AI 编程工具均通过 `install.sh` 一键安装支持。其他工具可以通过 `capture-generic.sh` 脚本手动集成。

### Q: 没有决策数据怎么办？

**A:** 使用 `npm run start:mock` 启动 Mock 模式，自动生成包含所有场景类型的模拟数据，让你体验全部功能。真实数据会在你使用 PI 工作时自动积累。

### Q: 如何减少存储占用？

**A:** 可以在可视化界面中删除不需要的会话（`DELETE /api/sessions/:id`），或直接清理 `~/.pi/decisions/` 下的旧日期目录。每个会话的 JSONL 文件通常只有几十 KB，即使长期使用也不会占用太多空间。

### Q: 可视化服务占用多少资源？

**A:** Express 服务端非常轻量，默认运行在 3141 端口。前端使用 React 19 + React Flow 12，对现代浏览器无压力。不使用时可以随时关闭。

### Q: 能不能同时监控多个项目？

**A:** 可以。用 `--port` 参数指定不同端口，`--source` 参数指定不同的数据目录，即可同时打开多个可视化实例。

### Q: 时间线回放卡顿怎么办？

**A:** 如果单个会话的节点数非常多（100+），可以尝试：1) 折叠不关注的子树；2) 使用时间线滑块跳到感兴趣的时间段。前端采用虚拟化渲染，正常情况下不会卡顿。
