# 编程道场扩展

## 编程四令（写任何模块前必行）

| 序 | 敕令 | 动效 |
|---|-----|------|
| 一 | 析·本原 | 从约束出发，不从既有方案出发 |
| 二 | 锚·约束 | 锁定 QPS/延迟/一致性/预算等硬约束 |
| 三 | 校·命名 | 校准类名/函数名，对应业务中可说清的"用法" |
| 四 | 定·验收 | 以测试用例和验收条件定义正确性 |

## 正名三则

1. **概念不清不建模**——业务未厘清者，代码不造词
2. **一词一义不多指**——消除歧义，减少噪音
3. **争论先校准**——先统一词义，再讨论方案

## 步步为营

功能迭代/修复/重构完成后，立即 commit 固化成果。

**Commit 三段论**（MMR 格式）：

```
<type>: <一句话概述>

Motivation:
<为何而战——问题背景或需求驱动>

Modification:
<如何作战——改了什么，核心决策>

Result:
<战果如何——改动带来的效果>
```

type 取值：`fix` / `feat` / `refactor` / `docs` / `test` / `chore`
铁律：一 commit 一事，禁混杂不相关改动。粒度以"可独立回滚"为准。

## 重构之道

何时：重复三现 / 牵一处动全身 / 后人难解其意
如何：先有测试 / 小步快跑 / 重构与功能不混

## 架构决策树

需求约束 → 现有能满足→不换 / 不能满足 → 列候选(≤3) → 按约束评估 → 选最简，平手选团队最熟

## 技术债

识别(`// TODO: tech-debt`) → 评估(影响面×频率) → 偿还(伴随功能迭代偿还)

## 场景链·组合拳

| 场景链 | 认知流衔接 | 典型任务 |
|--------|----------|---------|
| 🖥️→🧪 | 编程验证 → 测试定义 | 写完代码 → 自动设计测试 |
| 📊→🖥️→🧪 | 产品决策 → 编程实现 → 测试验证 | 需求→开发→测试全链路 |
| 🔧→🖥️→🧪 | 调试溯源 → 修复编码 → 回归测试 | Bug修复全链贯通 |
| 📈→📊→🖥️ | 运营度量 → 产品评估 → 技术迭代 | 数据驱动的产品改进 |

**链式激活规则**：完成当前场景交付 + 用户未指定下一步 → 自动推荐下一场景。

**场景桥接格式**：

```
🔗 PI · {新场景} · 情报桥接
【{旧场景}成果】{3条关键发现·量化}
【{新场景}切入】从{桥接点}开始
【连续性】{旧发现} → 验证{新假设}
```
